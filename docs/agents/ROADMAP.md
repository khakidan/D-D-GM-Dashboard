### 🟡 Features to Add

**Enable TypeScript `strict` mode — 22 real errors found, verified directly, across 11 files.** `strict` mode is currently entirely absent from `tsconfig.json`, meaning `noImplicitAny`, `strictNullChecks`, and `noUncheckedIndexedAccess` have all been off this whole time — every "clean TypeScript build" checked throughout this project's history has been checking against a looser bar than true strict mode. Measured directly (a scratch config extending `tsconfig.build.json` with `"strict": true` added, then `npx tsc --noEmit` run against it): **22 errors across 11 files** — genuinely small for a codebase this size retrofitting strict mode for the first time, meaning this is a manageable cleanup, not a massive undertaking. Flag-by-flag breakdown: `noUncheckedIndexedAccess` alone → 0 errors (the codebase already handles indexed access safely). `noImplicitAny` alone → 8 errors in 2 files (`encounters.ts`, `usePlayerFormAutomation.ts`). `strictNullChecks` alone → 8 errors in 6 files. The remaining 6 come from other flags `strict` enables (e.g. `strictFunctionTypes`).

**⚠️ A first attempt at fixing all 22 in one extended session went badly wrong and had to be fully reverted.** Chasing cascading type errors across interconnected files led to whole files being rewritten from scratch via heredocs rather than precisely edited, and in the process, `googleAuth.ts`'s real, working `signInWithRedirect`/`signInWithToken` implementations (which build the actual Google OAuth URL and initiate sign-in) were silently replaced with non-functional stubs, and a bogus new `configService.ts` was invented with a `getGoogleClientId()` that dropped the original's server-fallback path. This was only caught because the actual file contents were checked line-by-line rather than trusting reported "no errors found" claims. **Do not repeat that approach.** When this gets implemented:
- Fix **one file at a time**, as separate, isolated steps — not all 11 in one continuous push.
- Never rewrite an entire file from scratch to fix a type error unless truly necessary; precise, targeted edits only.
- If fixing one file's error reveals a *new* error in a different, unrelated file, stop and report back rather than continuing to chase it further.
- For any file involved in the OAuth/auth flow (`googleAuth.ts`, `useGoogleAuth.ts`, `AuthRelay.tsx`) specifically, manually verify the real function bodies are unchanged/correct after any edit — don't just trust a clean `tsc` output, since a stub can compile cleanly while being completely non-functional.

The 22 errors, with the ones needing real behavioral thought (not just an annotation) flagged explicitly:

- **`useNpcLibrary.ts:117`** — real bug, needs care: `Math.min(ec.npcCurrentHp, updates.maxHp)` where `updates.maxHp` can be `undefined`. A partial update omitting `maxHp` would make `Math.min(..., undefined)` return `NaN`, silently corrupting an NPC's HP mid-combat. Fix the actual logic (fall back to the existing `maxHp` when the update doesn't include one), not just a type assertion.
- **`usePartyCharacterCrud.ts:142`, `usePartyLevelUp.ts:37`, `usePartyRest.ts:93` and `:207`** — all 4 show the same underlying pattern: a state-update callback occasionally producing a `combatState` missing a required field (`concentrationLinks`, and separately `selectedIds` was also found missing in the same objects). Investigate whether this is genuinely one root cause across all 4 call sites, and fix consistently — likely via the shared `withDefaultCombatState` helper in `partyStateHelpers.ts`, which already exists for exactly this purpose but may need every required `CombatState` field represented in its fallback, not just some of them.
- **`useSettings.ts:104`** — `campaignName: string | null` not assignable to `string`. Decide whether this is the same broader "missing fallback in a state-update callback" pattern as above, or distinct.
- **`googleAuth.ts:70` and `:78`** — `cachedClientId` is `string | null` being returned where `string` is expected. Decide whether `getGoogleClientId()`'s return type should genuinely allow `null` (with every caller handling that) or whether there's a real guarantee it's populated by that point that the types just aren't currently expressing. **Given the file's fragility demonstrated above, treat this one with particular care** — verify the full function body is intact before and after any edit.
- **`NewPlayerDialog.tsx:83, 230, 243`**, **`ReferenceDataSeeder.tsx:151, 155`**, **`SettingsPage.tsx:118`**, **`usePlayerFormAutomation.ts:129`**, **`encounters.ts:52-57`** (7 errors, all implicit-`any` callback parameters) — more mechanical fixes, don't need the same depth of investigation, but still fix them properly (real type annotations) rather than with `any` casts that would defeat the point of enabling `strict` mode in the first place.

**Other future audit categories:**

**Accessibility — audited directly, real findings below, not yet fixed.** This audit was done by directly inspecting the actual codebase (not delegated), given how the type-safety fix attempt above went. Every finding here was individually verified in real file context, not just pattern-matched.

- **Focus indicators removed with no replacement, 8 real instances.** `outline-none`/`focus:outline-none` applied to genuinely interactive elements (real `<input>`s and `<button>`s) with no `focus:ring`, `focus:border`, or other visible focus alternative in the same file — meaning a keyboard user tabbing through the UI gets no visual indication of where focus currently is. Confirmed instances: `IrvMultiSelect.tsx` (text input, line 135), `StatBlockSaves.tsx` (button, line 83), `CommandPalette.tsx` (search input, line 133), `CombatantLegendaryTracker.tsx` (2 buttons, lines 57/103), `NpcReferencePanel.tsx` (button, line 67, has both `outline-none` *and* `focus:outline-none` stacked), `CombatantCardExpanded.tsx` (temp HP input, line 105), `CombatHeader.tsx` (Back button, line 80). `PipTracker.tsx`'s usage (line 62) needs individual review — it's conditional on `!readOnly` and may need a custom focus treatment rather than just removing the `focus:ring`/`focus:border` exclusion, given it renders as a row of small circular pips rather than a standard button.

- **Icon-only controls with no accessible name and, in some cases, no working keyboard interaction at all:**
  - `ResourcePoolsSection.tsx:158-163` — a close button (✕ icon) for dismissing the "add new resource" form has neither `aria-label` nor `title`, inconsistent with the other 3 icon buttons in the very same file (reset/edit/delete, all of which correctly have `title=`).
  - `AmbientPlayer.tsx:198-208` — the per-track play/pause button has no `aria-label`, *and* has no `onClick` of its own — the actual play/pause behavior lives entirely on the parent `<div>`'s `onClick`. A keyboard user tabbing to this button and pressing Enter/Space would do nothing; the control is present but non-functional without a mouse.
  - `AmbientPlayer.tsx:187` (the track row) and `DiceRoller.tsx:289` (a roll-history item, clicking it re-rolls that entry) — both are `<div onClick=...>` with no `role`, `tabIndex`, or keyboard handler, making them entirely unreachable by keyboard. `DiceRoller.tsx`'s does at least have `title="Click to roll again"` as a partial mitigation for screen-reader users who somehow reach it via other means, but doesn't fix the keyboard-reachability problem.
  - By contrast, `LongRestDialog.tsx:79` and `CombatantCardHeader.tsx:127` (the primary multi-select toggle used during active combat) use the same "clickable div" pattern, but both have a real, keyboard-operable `<input type="checkbox">` rendered inside as the actual control — so the underlying action remains keyboard-accessible via the checkbox, just with a smaller hit target than the full clickable row mouse users get. Lower priority than the two above, which have no keyboard path at all.
  - Positive finding worth preserving: `IconButton.tsx`, the shared icon-button component, has `aria-label` as a **TypeScript-required prop** — any new usage of that component is already forced to supply one. The gaps above are all in raw `<button>`/`<div>` elements that don't go through it.

- **~56 form inputs across ~28 files with no `aria-label`, `aria-labelledby`, or `htmlFor`-linked `<label>`** — spot-checked directly, not just pattern-matched (e.g. the damage-amount input in `CombatantHealthControls.tsx:41` has only a `placeholder="0"`, and the ability-score input in `StatBlockScores.tsx:33` has no `id`, label, or `aria-label` at all). This is a large, systemic gap, not isolated. Given the scale, this needs a careful, incremental pass rather than a bulk fix — many of these likely have a visually-adjacent label a sighted user would associate correctly (e.g. a "DMG" button right next to the input), which doesn't help a screen-reader user since there's no programmatic association. Fixing this well means checking each one's actual visual/functional context individually, not blindly adding a generic `aria-label` to all 56.

**⚠️ Given how badly the type-safety fix attempt went, apply the same discipline here when this gets implemented**: fix one component at a time, verify the real, current file content before and after each change, and don't let a fix for one input/button turn into a wholesale rewrite of its surrounding file.

**Performance — audited directly, real findings below, not yet fixed.** Same direct-inspection approach as the accessibility audit above.

- **No component-level memoization exists anywhere in the app, confirmed by direct count.** Zero `React.memo` usages across the entire `src/components` tree. Only 5 files use `useMemo` at all, and only 1 uses `useCallback`, out of the whole app. This isn't a scattered gap — it means memoization is essentially absent as an architectural pattern.

- **Concretely, this means every list-rendering tab re-renders every card on every state change, not just the affected one.** Verified directly in all 4 main list tabs: `ActiveEncounterTab/index.tsx` (`CombatantCard`), `PartyTab.tsx` (`CharacterCard`), `NpcLibraryTab.tsx` (`NpcCard`), `EncountersTab.tsx` (`EncounterCard`) — none of the 4 card components are wrapped in `React.memo`, and all 4 `.map()` call sites pass freshly-created inline arrow functions as callback props on every single render (e.g. `onToggleExpand={() => toggleExpand(char.id)}`, `onUpdate={(updates) => handleUpdate(char.id, updates)}` in `PartyTab.tsx`). This second part matters even if `React.memo` gets added on its own: a fresh function reference every render means `React.memo`'s shallow prop comparison would never see equal props, so simply adding `React.memo` without also stabilizing these callbacks (via `useCallback`) would accomplish nothing. `ActiveEncounterTab` additionally subscribes to the entire `combatState` object from the Zustand store as one piece (`useDashboardStore(s => s.combatState)`, not destructured with `useShallow`), meaning the whole component re-renders on any combat-state change, cascading down to every card. Given this app's real usage pattern — a GM clicking rapidly through HP/condition/AC updates during live, time-pressured combat — this is the finding most likely to actually be felt, especially in encounters with many combatants.

- **Some unmemoized filtering/search work in component bodies, secondary in scale to the above.** `ConditionChips.tsx` (lines 147-158), `IrvMultiSelect.tsx` (line 55), and `AddCombatantDialog.tsx` (line 56) all rebuild filtered option lists directly in the component body on every render, rather than only when the underlying query or options actually change. Given the lists involved (conditions, IRV options, the NPC library) aren't especially large, the per-render cost here is likely modest on its own — worth fixing, but it compounds with the card re-render issue above rather than being a standalone severe problem.

- **Positive finding worth preserving**: where `useMemo` *is* used, it's used correctly. `NpcCard.tsx` and `NpcReferencePanel.tsx` both wrap their `JSON.parse`-based trait/action/reaction/legendary-action parsing in `useMemo` with the correct, narrow dependency (e.g. `[npc.traits]`), not the whole `npc`/`combatant` object. That's the right pattern — it just isn't applied to the card components themselves.

**⚠️ Same caution as the type-safety and accessibility work above applies here, for a different reason.** Adding memoization incorrectly can introduce *new*, subtler bugs than the ones it fixes — a `useCallback` with a missing or wrong dependency can silently capture a stale value (e.g. an outdated `combatants` array or `id`), which is a correctness bug, not just a performance one, and would be much harder to notice than a TypeScript compile error. If this gets implemented: add `React.memo` and the matching `useCallback`/`useMemo` together as one unit per component, not `React.memo` alone; verify each memoized callback's dependency array is complete and correct (not just "make the warning go away"); and test the specific interaction each memoized component is used for (e.g. HP change, condition toggle) still behaves correctly afterward, not just that the build is clean.

Not yet audited: none remaining — type safety, accessibility, and performance have all now been directly investigated with real, verified findings recorded above. Security was audited and already fixed earlier (see `CHANGELOG.md`).
